# 家計簿管理アプリ 開発フロー
* **毎回必ず実行**
  * AGENT_GUIDELINE.mdを読み込む。

## 全体的な決め事 (General Decisions)

* **技術スタックと環境:** バックエンドは Node.js、フロントエンドは React.js の最新安定版を採用する。基本は Node/React 構成で進める。開発環境の統一と “動く環境” の再現性確保のため Docker コンテナを利用する。
* **アーキテクチャ指針:** DDD (ドメイン駆動設計) に基づき、ドメイン層・アプリケーション層・インフラ層の 3 層に責務を分離する。ビジネスロジックと技術依存コードを分離して保守性を高める。
* **データ永続化:** ローカル環境で動作する組み込みデータベース (SQLite など) でデータを永続化し、アプリを再起動しても入力データが失われないようにする。
* **基本機能要件:**

  * 月単位で「固定支出」「変動支出」「収入」を管理する。
  * 固定・変動支出は【支出名／タグ／金額】を入力し、各月ごと最大 20 件まで登録できる。
  * 固定支出は一度登録すればテンプレートとして次回以降選択できる。
  * 収入は定期収入 (月収) を初期登録し毎月自動セット、編集可。賞与・臨時収入は別枠で追加登録可。
  * 初期画面で選択年月の **収支比較グラフ** と **タグ別支出内訳グラフ** を表示する。
* **設定管理:** .env などの環境変数／設定ファイルで環境差分を吸収し、コードに環境値をハードコードしない。
* **コーディング規約:**

  * 命名は略語禁止 (40 文字超の場合のみ短縮を検討)。
  * クラス 200 行超・メソッド 50 行超になったらリファクタリングする。
  * クラス間は疎結合を保ち、共通処理は重複させず 1 箇所に集約 (DRY 原則)。
  * 実装方法を選択する際は処理速度と保守性を最優先し、その理由をコメントに明記する。
* **テスト方針:** すべての機能に対して単体／結合テストを作成し、少なくとも C2 (パスカバレッジ) を達成する。CI で自動テストを実行する。

## Docker コンテナ構築

Dockerfile と docker-compose.yml を用意し、`docker-compose up --build` で開発環境を起動できるようにする。アプリは Node 20 イメージでビルドし、ポート 3000 を公開する。

## Phase 1: 要件分析 (Requirement Analysis)

1. 家計簿に必要な機能要件を洗い出し、漏れを確認する。
2. 不明点を関係者にヒアリングして解消し、要件仕様を確定する。
3. MVP (Minimum Viable Product) と将来拡張範囲を明確化し、スコープを定義する。
4. 要件仕様を文書化し、本ドキュメント (AGENT.md) および要件定義書として共有する。

## Phase 2: ワークフロー構造設計 (Workflow & Structural Design)

1. 画面遷移・操作フローをワイヤーフレームで設計する。
2. ドメインモデル (Expense / Income / Tag など) と月次家計簿オブジェクトを設計する。
3. バックエンド API (REST もしくは GraphQL) とフロントエンドのコンポーネント構成を設計する。
4. データベーススキーマ (Expenses / Incomes / FixedExpenseTemplates) を設計する。
5. システム構成図・シーケンス図を作成し、各層の責務と依存関係を明確化する。
6. 性能要件・エラーハンドリング・ログ設計などの非機能要件を整理する。

## Phase 3: 共通処理の識別と外部化 (Identify & Modularize Common Functionality)

1. 重複しそうな処理 (金額合計計算、タグ別集計、入力バリデーション等) を洗い出す。
2. ユーティリティ関数／サービスクラスとして共通処理をモジュール化する。
3. React では共通 UI コンポーネント (フォーム、モーダル、ボタン等) を作成し再利用性を高める。
4. DB 接続・設定読み込み・ログ出力などインフラ処理もユーティリティ化し、各層に跨る重複を排除する。
5. 外部ライブラリ (バリデーションライブラリ、ORM など) を活用し、独自実装の範囲を最小化する。

## Phase 4: ドラフト作成 (Draft Implementation)

1. プロジェクト初期化 (Node/React/TypeScript・Docker セットアップ) を行う。
2. ドメインモデルと単体テストを実装する。
3. インフラ層 (DB 接続・リポジトリ) を実装し、マイグレーションスクリプトを用意する。
4. アプリケーション層のサービス／ユースケースを実装する。
5. Express (or Fastify) コントローラと API ルーティングを実装する。
6. React のページ／コンポーネントを実装し、API と連携する。
7. 単体テストを追加実装し、すべてグリーンになるまで修正する。

## Phase 5: レビューと確認 (Review & Verification)

1. コードレビューを実施し、規約順守・設計整合・バグ有無をチェックする。
2. 手動動作確認で基本シナリオを検証し、問題点を修正する。
3. パフォーマンスと UX を確認し、必要に応じて最適化する。
4. ドキュメント (AGENT.md / README) を最新化し、セットアップ手順を確認する。
